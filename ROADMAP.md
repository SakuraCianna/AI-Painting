# AI Painting 长期产品路线图

> 日期: 2026-06-12
> 当前阶段: MVP 后产品化扩展
> 核心约束: 用户不能使用鼠标或键盘绘图, 只能通过语音完成创作、编辑、确认和导出

## 1. 产品北极星

AI Painting 的长期目标不是做一个普通文生图工具, 而是做一款“语音优先的可编辑绘图工作台”。

用户应该可以用自然语言完成这些事:

- 从一句描述生成结构化画面
- 继续用语音精确修改局部对象
- 让系统在不确定时主动追问
- 用语音管理图层、对象、样式、导出和版本
- 在需要丰富视觉时生成背景或素材图层

商业可用的判断标准:

- **可控**: 每个对象可命名、可选择、可修改、可撤销
- **稳定**: ASR、NLU、执行引擎都有降级和错误解释
- **低延迟**: 简单指令快速反馈, 复杂指令分阶段反馈
- **可评测**: 每类语音指令都有评测集和成功率指标
- **可扩展**: 新图形、新模型、新 Provider 可以通过清晰接口接入

## 2. 产品原则

### 2.1 矢量优先, 生图增强

默认绘图路径保持为:

```txt
语音 -> ASR -> 指令理解 -> 结构化计划 -> 矢量对象 -> 可编辑画布
```

生图模型只作为增强能力:

- 生成背景图层
- 生成纹理和贴纸素材
- 生成参考图
- 将生成图片作为可移动、可裁剪、可替换的图片对象插入画布

不把生图模型作为核心绘图引擎, 因为它不利于精确编辑、结构化撤销和低延迟交互。

### 2.2 语音是唯一创作输入

可以有 icon 按钮用于开始录音、暂停、重试、导出等系统控制, 但不能提供鼠标拖拽绘图、键盘快捷绘图或隐藏的手工编辑路径。

### 2.3 复杂指令先拆解, 再执行

复杂描述不能直接变成一坨不可解释的结果。系统需要生成可读的执行计划, 例如:

```json
{
  "intent": "compose_scene",
  "steps": [
    "创建背景",
    "添加主体对象",
    "添加局部细节",
    "应用统一风格",
    "检查遮挡和对齐"
  ]
}
```

执行前后都要能解释:

- 听到了什么
- 理解成什么
- 准备画哪些对象
- 哪些步骤已完成
- 哪些步骤需要用户确认

## 3. 阶段路线

### 阶段 A: 语音绘图内核 2.0

目标: 从基础图形工具升级为结构化画布引擎。

关键能力:

- 对象命名, 例如“把这棵树命名为左边的大树”
- 多对象选择, 例如“选中所有蓝色圆形”
- 语义选择, 例如“把房子的窗户都变大”
- 图层, 例如“把太阳放到云后面”
- 分组, 例如“把房子和树组成一个小院子”
- 旋转、缩放、对齐、分布
- path、polygon、bezier、freeform path
- 渐变、阴影、透明度、虚线、圆角
- 对象标签 `semantic_tag`, 例如 `house.window`, `tree.leaf`

技术任务:

- 扩展 `drawing_objects` 的对象模型
- 增加 `group_id`、`layer_id`、`transform_json`、`semantic_tags_json`
- 增加可回滚的复合操作
- 增加对象查询 DSL
- 增加复杂对象的 SVG 渲染

验收标准:

- 能用语音画出包含 20 个以上对象的房子、树、道路、太阳和云朵场景
- 能对场景中某个语义子对象继续修改
- 撤销和恢复能按用户语义工作, 不是只能一步步机械回退

### 阶段 B: 复杂指令规划器 2.0

目标: 让系统能把自然语言描述拆成稳定、可校验、可执行的画面计划。

关键能力:

- 引入 `ScenePlan`
- 支持 `compose_scene`、`edit_scene`、`refine_object`、`generate_asset`
- 规则解析处理高频短指令
- LLM 处理复杂构图和模糊描述
- 低置信度时先追问, 不盲目执行
- 计划执行前做 schema 校验、尺寸校验、对象数量限制和安全检查

示例能力:

- “画一个温馨的小屋, 左边有两棵树, 右边有一条弯曲小路, 天空有三朵云”
- “把画面改成夜晚, 保留房子的形状, 给窗户加暖黄色灯光”
- “把树稍微往后放, 不要挡住门”

技术任务:

- 新增 `ScenePlan` schema
- 为 MiMo 规划器增加严格 JSON 输出模板
- 增加计划修复器, 自动补全缺失字段
- 增加计划解释文本
- 增加复杂指令评测集

验收标准:

- 100 条复杂中文指令的计划生成成功率达到 85% 以上
- 简单指令不依赖 LLM, 保持低延迟
- 所有 LLM 计划都必须经过后端校验后才能执行

### 阶段 C: 语音交互与低延迟体验

目标: 让用户感觉是在和画布对话, 不是录一段音频等结果。

关键能力:

- 流式 ASR
- 语音活动检测 VAD
- 局部 transcript 展示
- 端点检测和打断
- 指令确认和二次追问
- TTS 简短反馈
- 语音命令上下文, 例如“把它再小一点”

技术任务:

- 保留小米 ASR、本地 ASR、Web Speech API 兜底
- 增加流式 ASR Provider 接口
- 增加语音延迟埋点: `vad_ms`、`asr_ms`、`plan_ms`、`execute_ms`、`tts_ms`
- 增加真实音频评测集
- 增加领域词表, 例如颜色、形状、布局、画风、导出格式

验收标准:

- 简单指令从停止说话到画布变化控制在 800ms 到 1500ms
- 复杂指令 2s 内给出“正在拆解”的反馈
- ASR 失败时有明确兜底路径和用户可理解提示

### 阶段 D: 生成式素材和背景图层

目标: 在不破坏可编辑性的前提下, 引入生图模型提升画面表现力。

关键能力:

- 语音生成背景图
- 语音生成贴纸或素材
- 把图片作为 `image` 对象加入画布
- 图片对象可移动、缩放、裁剪、替换
- 生成失败时不影响矢量画布
- 支持 Provider 抽象, 先接官方文档明确的图像 API, 后续再接小米生图能力

技术任务:

- 新增 `image_assets` 表
- 新增 `image` 绘图对象类型
- 新增 `ImageProvider` 接口
- 新增素材生成任务状态: pending、running、succeeded、failed
- 新增内容安全和尺寸限制

验收标准:

- “生成一张水彩风森林背景”能创建图片图层
- “把背景换成夜晚星空”不会破坏前景矢量对象
- 图片生成任务失败时画布状态不回滚、不污染历史

### 阶段 E: 专业画布底座

目标: 从简单 SVG 画布升级到可支撑复杂作品的画布系统。

可选路线:

1. 继续自研 SVG/Canvas 混合引擎
2. 调研并逐步迁移到 tldraw 或 Excalidraw 这类成熟画布底座

短期建议:

- 不立即迁移底座
- 先把后端对象模型和操作日志设计成独立领域模型
- 等对象模型稳定后, 再评估前端渲染层是否切换

关键能力:

- 无限画布或大画布
- 缩放和平移, 但只能通过语音触发
- 对齐、分布、吸附
- 图层管理
- 组件库和复用素材
- 版本快照和恢复
- 大作品性能优化

验收标准:

- 200 个对象以内保持流畅渲染
- 作品可导出 SVG、PNG 和项目 JSON
- 画布状态可以完整恢复

### 阶段 F: 产品级工作流

目标: 让工具从 Demo 变成可被真实用户持续使用的产品。

关键能力:

- 作品列表和搜索
- 版本历史
- 模板库
- 风格预设
- 多语言界面
- 可访问性检查
- 本地优先保存
- 可选账号和云同步
- 作品导入导出
- 用户指令历史复用

商业化前置能力:

- 错误监控
- 语音识别和规划耗时统计
- Provider 成本统计
- API Key 安全管理
- 内容安全和滥用防护
- 隐私说明和数据保留策略

## 4. 指标体系

### 4.1 语音识别

- ASR 成功率
- 字错误率 WER
- 领域词识别率
- 端点检测准确率
- 静音误触发率
- 平均 ASR 延迟

### 4.2 指令理解

- 意图识别准确率
- 计划生成成功率
- 需要追问比例
- 错误执行率
- 用户修正次数

### 4.3 绘图执行

- 简单指令执行延迟
- 复杂计划执行延迟
- 单次计划对象数量
- 撤销恢复成功率
- 画布渲染耗时

### 4.4 产品体验

- 首次成功绘图时间
- 每个作品平均语音轮次
- 导出成功率
- 用户放弃率
- 生成式素材成功率

## 5. 近期优先级

### P0

- 建立复杂指令评测集
- 增加对象命名和语义标签
- 增加多对象选择和图层
- 增加 `ScenePlan` schema

### P1

- 增加 path、polygon、bezier
- 增加分组和复合撤销
- 增加低置信度追问机制
- 增加 ASR 和规划延迟指标

### P2

- 增加图片对象和素材库
- 增加生图 Provider 抽象
- 增加导出 SVG 和项目 JSON
- 调研画布底座迁移成本
