# 纯语音控制绘图工具需求文档

> 状态: MVP 基线与持续演进需求
> 日期: 2026-06-12
> 技术栈约束: Python 3.12.10 虚拟环境 + React 前后端分离 + SQLite
> 当前交付范围: 需求、实现、设计对照、差距评估和后续路线持续维护

## 1. 背景

本项目需要开发一款纯语音控制的绘图工具。用户不能通过鼠标或键盘完成绘图创作, 应通过自然语言语音指令完成画布创建、图形绘制、样式调整、对象编辑、撤销恢复、保存导出等操作。

由于浏览器语音识别能力存在兼容性差异, 语音输入层必须做成可替换模块。当前实现采用小米 MiMo ASR、本地 ASR 和 Web Speech API 三层降级方案, 其中本地 ASR 已提供 Qwen3-ASR Windows 11 服务脚手架, 但真实 ASR 稳定性仍需要后续评测。

当前阶段不是纯需求草案阶段, 而是以 MVP 为基线继续产品化。实现进度、未完成项和差距说明分别维护在 [设计文档.md](设计文档.md)、[ROADMAP.md](ROADMAP.md) 和 [docs/status/voice-drawing-gap-analysis.md](docs/status/voice-drawing-gap-analysis.md)。

## 2. 产品目标

1. 用户可以只通过语音完成一幅基础矢量风格画作。
2. 系统可以把中文自然语言指令解析成确定的绘图操作。
3. 系统可以容忍同义词、口误、参数缺失和复杂复合指令。
4. 简单指令在语音识别完成后应快速反映到画布上。
5. 复杂指令应能拆解为多个可追踪、可撤销的绘图步骤。
6. 作品、操作历史和语音指令日志应持久化到 SQLite。
7. 后续实现完成后, 需要额外提交设计文档, 记录计划支持、最终实现和未完成原因。

## 3. 核心约束

1. 用户绘图创作过程禁止依赖鼠标和键盘。
2. 浏览器首次麦克风授权可能需要系统或浏览器权限确认, 该动作属于运行环境限制, 不计入应用内绘图操作。
3. 应用内不得提供鼠标拖拽绘图、键盘快捷键绘图等绕过纯语音目标的核心路径。
4. 可以保留只读视觉反馈, 例如画布、识别文字、执行状态、错误提示和操作历史。
5. 所有关键命令都应有语音反馈或可视化反馈, 避免用户不知道系统是否听懂。
6. 不默认保存原始音频, 只保存识别文本、解析结果、执行结果和错误信息。
7. 不在需求阶段绑定任何付费 ASR、LLM 或云模型供应商。

## 4. 用户角色

### 4.1 主要用户

想通过语音完成绘图创作的普通用户, 包括不方便使用鼠标键盘的用户。

### 4.2 次要用户

开发者和测试人员, 需要复现语音指令、检查解析结果、查看操作日志和定位响应延迟问题。

## 5. 典型使用流程

1. 用户打开应用。
2. 用户授权麦克风。
3. 用户说“新建一张横向画布, 白色背景”。
4. 系统创建画布并语音反馈“已创建横向白色画布”。
5. 用户说“画一个蓝色圆形在中间, 半径一百”。
6. 系统解析为绘制圆形操作并渲染到画布。
7. 用户说“再画三颗黄色星星, 从左到右变小”。
8. 系统拆解为三个星形绘制步骤并依次执行。
9. 用户说“把刚才的圆改成浅蓝色, 加粗边框”。
10. 系统定位最近的圆形对象并修改样式。
11. 用户说“撤销上一步”。
12. 系统回滚最近一次操作。
13. 用户说“保存作品, 名字叫夏日练习”。
14. 系统保存作品和操作历史到 SQLite。
15. 用户说“导出为 PNG”。
16. 系统生成导出文件并提示完成。

## 6. 指令能力需求

### 6.1 画布控制

必须支持:

1. 新建画布, 例如“新建画布”“创建一张 16 比 9 的画布”。
2. 设置画布尺寸, 例如“宽一千二, 高八百”。
3. 设置背景颜色, 例如“背景改成米白色”。
4. 清空画布, 需要二次确认。
5. 保存作品。
6. 打开最近作品。
7. 导出 PNG。

增强支持:

1. 导出 SVG。
2. 复制当前作品为新版本。
3. 切换深色或浅色界面。

### 6.2 基础绘图

必须支持:

1. 点、线、箭头。
2. 矩形、圆形、椭圆、三角形。
3. 文本对象。
4. 简单自由曲线, 通过语音描述路径, 例如“画一条从左下到右上的弯曲线”。
5. 常见颜色名称, 例如红色、蓝色、浅蓝色、黑色、白色、透明。
6. 常见位置描述, 例如中间、左上角、右侧、底部、靠近标题。
7. 常见尺寸描述, 例如大一点、小一点、半径一百、宽三百。

增强支持:

1. 星形、心形、云朵、多边形。
2. 渐变填充。
3. 阴影、虚线、透明度。
4. 基于网格的对齐, 例如“和左边矩形顶部对齐”。

### 6.3 对象编辑

必须支持:

1. 选择最近创建的对象, 例如“选中刚才那个圆”。
2. 按类型选择对象, 例如“选中所有蓝色圆形”。
3. 移动对象, 例如“把它向右移动二十像素”。
4. 缩放对象, 例如“把它放大一倍”。
5. 修改填充色、边框色、线宽和文本内容。
6. 删除对象, 需要可撤销。
7. 撤销和恢复。

增强支持:

1. 旋转对象。
2. 编组和取消编组。
3. 图层上移、下移、置顶、置底。
4. 按语义命名对象, 例如“把这个圆命名为太阳”。

### 6.4 复杂指令拆解

必须支持把一句复杂指令拆成多个操作:

1. “画一个房子, 红色屋顶, 蓝色门, 两扇窗户”。
2. “在左边画三棵树, 从近到远逐渐变小”。
3. “把所有蓝色图形改成绿色, 然后整体向上移动一点”。

复杂指令执行要求:

1. 解析阶段生成步骤计划。
2. 简单低风险操作可直接执行。
3. 高风险操作需要确认, 例如清空画布、删除多个对象、覆盖保存。
4. 每个拆解步骤都写入操作历史。
5. 任一步失败时, 系统应停止后续步骤并提示失败原因。
6. 用户可以说“取消刚才的计划”中止未执行步骤。

### 6.5 纠错与容错

必须支持:

1. 同义词, 例如“撤销”“退回一步”“取消上一步”。
2. 口语量词, 例如“往右一点”“大一些”“三个小圆点”。
3. 参数补全, 缺少颜色时使用当前画笔颜色, 缺少位置时使用画布中心。
4. 歧义澄清, 例如“你是指最近的圆还是所有圆?”。
5. 识别纠错, 例如用户说“不对, 我说的是浅蓝色”。
6. 置信度不足时不执行 destructive 操作。

增强支持:

1. 方言或中英混合命令。
2. 用户自定义别名, 例如“我的常用蓝”。
3. 根据历史习惯推荐默认参数。

### 6.6 语音反馈

必须支持:

1. 麦克风监听状态提示。
2. 识别中文本提示。
3. 执行完成提示。
4. 失败原因提示。
5. 需要确认时的明确提问。

增强支持:

1. 使用浏览器语音合成读出执行结果。
2. 支持“安静模式”, 只显示文字反馈。

## 7. 性能与延迟需求

### 7.1 简单指令

目标:

1. 语音识别完成到画布更新, P75 不超过 800ms。
2. 语音识别完成到画布更新, P95 不超过 1500ms。
3. 操作历史写入不阻塞画布即时反馈。

### 7.2 复杂指令

目标:

1. 复杂指令解析到展示执行计划, P75 不超过 2000ms。
2. 第一条绘图操作开始执行, P75 不超过 3000ms。
3. 超过 3000ms 时必须显示“正在拆解指令”状态。

### 7.3 日志指标

需要记录:

1. `speech_started_at`
2. `speech_ended_at`
3. `transcript_received_at`
4. `parse_started_at`
5. `parse_finished_at`
6. `execution_started_at`
7. `execution_finished_at`
8. `render_finished_at`
9. 指令置信度
10. 操作成功或失败原因

## 8. 技术架构需求

### 8.1 前端

技术约束:

1. React。
2. 前端负责麦克风状态、语音识别适配、画布渲染、指令状态反馈和操作历史展示。
3. 绘图对象建议以矢量对象模型维护, 再渲染到 Canvas 或 SVG。
4. 不允许把鼠标和键盘作为核心绘图输入。

建议模块:

1. `VoiceController`: 管理语音监听、开始、停止、错误状态。
2. `CommandPanel`: 展示识别文本、解析结果、确认问题和执行状态。
3. `DrawingCanvas`: 渲染作品。
4. `OperationTimeline`: 展示可撤销操作历史。
5. `useVoiceRecognition`: 封装浏览器或后端语音识别差异。
6. `useCommandExecution`: 连接解析结果与画布操作。

### 8.2 后端

技术约束:

1. Python 3.12.10。
2. 使用虚拟环境管理依赖。
3. 前后端分离, 后端提供 API。
4. SQLite 持久化作品、版本、操作历史和语音指令日志。

建议模块:

1. `api`: REST 或 WebSocket 接口。
2. `command_parser`: 把识别文本解析为绘图操作。
3. `command_planner`: 拆解复杂指令。
4. `drawing_engine`: 校验操作并更新作品状态。
5. `repositories`: 访问 SQLite。
6. `schemas`: 请求、响应和操作类型定义。
7. `services`: 作品保存、导出、日志统计。

### 8.3 通信方式

必须支持:

1. 创建作品。
2. 获取作品。
3. 保存作品。
4. 提交语音识别文本并返回解析结果。
5. 执行绘图操作。
6. 获取操作历史。

建议支持:

1. WebSocket 用于低延迟指令状态推送。
2. REST 用于作品管理、导出和历史查询。

## 10. API 需求草案

### 10.1 作品管理

1. `POST /api/artworks`: 创建作品。
2. `GET /api/artworks`: 获取作品列表。
3. `GET /api/artworks/{id}`: 获取作品详情。
4. `PATCH /api/artworks/{id}`: 修改作品标题或画布属性。
5. `POST /api/artworks/{id}/save-version`: 保存版本。

### 10.2 指令解析与执行

1. `POST /api/commands/parse`: 输入识别文本, 返回操作计划。
2. `POST /api/artworks/{id}/operations`: 执行单个操作。
3. `POST /api/artworks/{id}/plans`: 执行复杂操作计划。
4. `POST /api/artworks/{id}/undo`: 撤销。
5. `POST /api/artworks/{id}/redo`: 恢复。

### 10.3 导出

1. `POST /api/artworks/{id}/export/png`: 导出 PNG。
2. `POST /api/artworks/{id}/export/svg`: 导出 SVG, 增强能力。

## 11. 操作类型需求

基础操作类型:

1. `create_canvas`
2. `update_canvas`
3. `add_object`
4. `update_object`
5. `delete_object`
6. `select_object`
7. `move_object`
8. `scale_object`
9. `set_style`
10. `add_text`
11. `undo`
12. `redo`
13. `save_artwork`
14. `export_artwork`

操作要求:

1. 每个操作必须有唯一 ID。
2. 每个可改变作品状态的操作必须能生成撤销数据。
3. 执行前必须校验目标对象是否存在。
4. 执行后必须返回画布最新状态或增量变更。
5. 执行失败不能污染作品状态。

## 12. 指令解析需求

解析流程:

1. 原始识别文本。
2. 文本规范化, 处理空格、标点、数字中文转阿拉伯数字、同义词。
3. 意图识别, 例如绘制、选择、修改、删除、保存、导出。
4. 参数抽取, 例如对象类型、颜色、位置、尺寸、数量。
5. 歧义判断。
6. 生成操作计划。
7. 风险判断。
8. 返回可执行计划或澄清问题。

必须内置的词表:

1. 颜色词表。
2. 形状词表。
3. 方位词表。
4. 尺寸词表。
5. 操作动词词表。
6. 确认和取消词表。
7. 撤销和恢复词表。

## 13. 安全与隐私需求

1. 不默认保存原始音频。
2. 语音日志只保存文本和解析结果。
3. 如果后续接入第三方 ASR 或 LLM, 必须在设计文档中说明数据发送范围、密钥配置方式和隐私风险。
4. `.env` 不得提交密钥。
5. destructive 操作必须支持撤销或二次确认。
6. 清空画布、删除多个对象、覆盖保存需要确认。

## 14. 可访问性需求

1. 主要状态必须可视化, 例如“正在听”“正在解析”“执行成功”“需要确认”。
2. 关键反馈应支持语音播报。
3. 颜色不能作为唯一状态表达。
4. 文本提示需要有足够对比度。
5. 画布对象被选中时需要有明显边框或轮廓。
6. 对语音识别失败应提供可重复指令, 例如“请再说一遍”。

## 19. 已知风险

1. 浏览器语音识别兼容性有限, 需要预留后端 ASR 方案。
2. 中文自然语言绘图指令存在歧义, 需要通过澄清问题和默认参数降低失败率。
3. 复杂指令如果一次性执行过多步骤, 可能导致用户难以理解结果, 需要操作计划和撤销能力。
4. 纯语音模式下, 用户无法直接拖拽调整细节, 需要更细的语音编辑指令弥补。
5. PNG 导出如果基于前端画布实现, 需要保证后端保存状态和前端渲染状态一致。
6. SQLite 外键约束需要在连接层明确开启, 否则关系约束可能不生效。

5. SQLite 官方文档, 用于确认外键约束和 JSON 字段处理注意事项。
