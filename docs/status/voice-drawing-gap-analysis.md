# 纯语音绘图目标差距评估

> 日期: 2026-06-12
> 分支: codex/AI-Painting
> 依据: 当前仓库代码、测试、README、需求文档、设计文档和路线图

## 1. 总体结论

当前项目已经达到“可演示的纯语音绘图 MVP”, 但距离真正商业可用的“用户不能使用鼠标或键盘, 仅通过语音完成绘图创作”仍有明显距离。

按原始要求拆分后, 当前成熟度可以这样判断:

| 维度 | 当前成熟度 | 说明 |
| --- | --- | --- |
| 纯语音绘图闭环 | 约 70% | 已能通过语音文本生成对象、编辑、撤销、恢复、保存、导出, 前端没有鼠标拖拽绘图工具 |
| 指令理解准确性与容错 | 约 58% | 高频规则指令较稳定, 复杂场景已能追问或交给 Drawing Agent 规划, 但缺少大规模真实语音评测 |
| 响应延迟控制 | 约 50% | 已记录 ASR、规划、执行和端到端指标, 并新增命令日志 P50/P75/P95 统计接口, 但没有真实音频样本和流式 ASR |
| 复杂指令拆解执行 | 约 60% | 已支持房子、人物头像、小屋场景、语音流程图、星星、批量改色移动和部分路径场景, 但开放式多主体构图仍以追问和规划器兜底为主 |
| 商业级产品可用性 | 约 42% | 核心架构方向正确, 已有图片对象、生图 Provider 和图生图精修脚手架, 但缺少真实 ASR/图像模型验证、无鼠标端到端测试、项目导入导出和稳定性指标 |

因此, 当前不是“只差 UI 打磨”的阶段, 而是已经完成基础闭环, 接下来需要把语音识别、复杂规划、语义对象模型和可评测指标做扎实。

## 2. 已经达成的要求

### 2.1 纯语音绘图路径

- 前端只提供画布、状态、识别文本、执行计划、延迟指标和操作历史。
- 没有鼠标拖拽绘图、键盘快捷绘图或隐藏手工绘图工具。
- 通过语音指令可以创建画布、添加图形、修改样式、移动、缩放、撤销、恢复、保存和导出。
- 浏览器麦克风授权和开始录音按钮属于系统控制, 不作为绘图创作入口。

### 2.2 ASR 多层降级

- 默认优先使用后端小米 MiMo ASR。
- 本地 ASR 支持 HTTP 服务和命令模板两种接入方式。
- 已提供 Windows 11 本地 Qwen3-ASR 服务脚手架。
- 后端 ASR 不可用时, 前端可降级到 Web Speech API。
- 前端会显示当前 ASR Provider 和最近一次 ASR 延迟。

### 2.3 指令解析与规划

- 高频短指令走本地规则解析, 延迟低、可测试、可解释。
- 复杂指令可调用小米 MiMo-v2.5-Pro 生成受控 JSON 操作计划。
- 多主体复杂场景在规则层不会被误执行为半成品, 会进入追问或规划路径。
- `CommandPlan` 已包含置信度、计划来源、解释文本、确认状态和 `ScenePlan` 骨架。

### 2.4 绘图能力

- 已支持圆形、矩形、椭圆、三角形、线条、箭头、星形、文本、多边形、路径和贝塞尔曲线。
- 已支持房子、人物头像、温馨小屋、语音流程图、太阳、云、弯曲小路等第一批组合场景。
- 已支持对象命名、图层、分组 ID、语义标签和 transform 预留字段。
- 已支持按图层和语义标签执行部分批量编辑。
- 已支持 `image` 图片对象和文字转图片 Provider 脚手架。
- 已支持“精修我的图片”这类图生图精修入口, 前端会上传当前画布截图, 后端将结果作为覆盖式图片层保存。
- 已支持语义对象形状替换和空间语义筛选, 例如“把窗户改成圆形”“把左边窗户改大一点”。

### 2.5 历史、撤销和安全确认

- SQLite 保存作品、对象、版本、操作历史和语音指令日志。
- 多步骤计划作为事务执行, 失败时回滚本次计划。
- “清空画布”保留二次确认, 确认后会真正清空对象, 并支持撤销恢复。
- 操作日志记录规划、执行和总耗时指标。

## 3. 主要差距

### 3.1 真正无鼠标体验仍未完成验收

当前 UI 不提供鼠标绘图路径, 但仍缺少端到端自动化验收来证明用户可以完全不使用鼠标或键盘完成一幅作品。浏览器麦克风授权、点击开始监听和导出下载也需要明确作为系统控制边界处理。

下一步需要补 Playwright 验收: 使用模拟语音文本或模拟 ASR 返回, 只通过语音命令 API 推动画布变化, 验证完整作品创建、编辑、撤销、恢复和导出路径。

### 3.2 ASR 稳定性还没有真实数据

当前完成的是接入链路和降级架构, 不是 ASR 稳定性结论。

未验证项:

- 小米 MiMo ASR 在真实账号、真实麦克风、噪声和长句环境下的准确率。
- Qwen3-ASR 本地模型在 Windows 11 上的实际安装、模型下载、显存或内存占用和延迟。
- Web Speech API 在目标浏览器和中文环境下的可用性。
- 三层 ASR 降级在真实网络抖动下的用户感知。

### 3.3 延迟指标有埋点和统计接口, 但没有 SLO

当前已记录 ASR、规划、执行和端到端耗时, 并提供 `GET /api/metrics/latency` 汇总最近命令日志的平均值、P50、P75、P95 和最大耗时。但还没有:

- 本地或 CI 的延迟基线。
- 真实音频样本回放。
- “超过 3 秒时显示正在拆解”的完整前端验收。
- 流式 ASR 和流式规划反馈。

### 3.4 复杂图形能力仍偏模板化

目前复杂指令主要覆盖一批高频模板, 例如房子、人物头像、小屋场景、语音流程图、星星、太阳和云。商业可用需要支持更开放的自然语言描述, 并能稳定产出结构化对象, 而不是只给出追问。

需要补齐:

- 更强的 `ScenePlan` schema 和计划修复器。
- 复杂场景评测集从当前第一版扩到 50 到 100 条。
- 对象查询 DSL, 支持“左边第二棵树”“屋顶下面的门”“所有暖色小物件”等引用。
- 组级撤销、组级移动、组级缩放和局部语义编辑。

### 3.5 可编辑画布能力不够完整

距离产品级绘图工具还缺:

- 旋转、对齐、分布、置顶、置底。
- 完整编组和取消编组。
- 渐变、阴影、透明度、虚线和圆角。
- 图片对象、素材图层和背景图层。
- 后端 SVG 导出、项目 JSON 导入导出。
- 版本历史 UI 和作品列表。

## 4. 为什么暂不把生图模型作为核心方案

生图模型可以提升画面表现力, 但不适合作为当前核心绘图引擎。

原因:

- 生图结果不天然可编辑, 难以做到“把左边第二扇窗户变大一点”。
- 难以稳定生成可撤销、可重放、可解释的操作历史。
- 延迟、成本和内容安全都更难控制。
- 原始需求强调“绘图创作”和“复杂指令拆解与执行能力”, 更适合结构化矢量对象模型。

推荐路线是“矢量优先, 生图增强”:

1. 语音生成结构化矢量场景。
2. 生图模型只用于背景、纹理、贴纸或参考图。
3. 生成图片作为 `image` 对象插入画布, 仍然可以用语音移动、缩放、替换和删除。
4. 当前已完成 `placeholder/http` 文生图 Provider 和 OpenAI 兼容图生图精修 Provider 第一版, 真实模型效果仍需后续验证。

## 5. 推荐近期目标

### 阶段 1: 真实 ASR 与延迟评测

- 已新增 `backend\evaluate_asr_samples.py` 和 ASR 样本 manifest 脚手架。
- 下一步建立 30 到 50 条中文音频样本。
- 分别评测小米 ASR、本地 Qwen3-ASR、Web Speech API 的成功率和延迟。
- 使用 `GET /api/metrics/latency` 汇总命令日志, 并用 ASR 样本评测输出 CER、成功率和延迟。
- 前端明确展示正在录音、正在识别、正在规划、正在执行。

### 阶段 2: 复杂规划器 2.0

- 扩展 `ScenePlan` 为可执行场景计划。
- 为 Drawing Agent 增加严格 SceneGraph JSON 模板和计划修复器。
- 增加复杂指令评测集到 100 条。
- 所有复杂计划先校验对象数量、尺寸、坐标、风险等级, 再执行。

### 阶段 3: 复合撤销与分组

- 引入复合操作 ID。
- 支持“撤销刚才那组房子”这类语义撤销。
- 完整实现 group 创建、组级移动、组级缩放、取消分组。
- 让批量操作在历史里以用户语义呈现, 而不是只看底层步骤。

### 阶段 4: 产品级无鼠标验收

- 增加前端 Playwright 测试。
- 模拟 ASR 返回, 验证只通过语音命令完成作品。
- 加入无键盘快捷绘图入口检查。
- 加入画布非空、对象数量、导出路径和历史记录断言。
