# 纯语音控制绘图工具设计文档

> 日期: 2026-06-12
> 分支: codex/AI-Painting
> 实现阶段: MVP 后产品化扩展

长期产品目标和商业级演进路线见 [ROADMAP.md](ROADMAP.md)。

## 1. 设计目标

第一阶段目标是打通“语音文本输入 -> 中文指令解析 -> 绘图操作执行 -> SVG 画布渲染 -> SQLite 持久化 -> 撤销恢复”的最小闭环。

当前阶段采用可降级外部能力: 小米 MiMo ASR 和 MiMo-v2.5-Pro 用于提升识别与复杂指令规划稳定性, 本地规则解析、本地 ASR 适配和 Web Speech API 作为降级路径, 避免外部服务不可用时完全阻断核心绘图闭环。

## 1.1 当前离原始要求的距离

当前项目已经达到可演示 MVP, 但还没有达到商业可用的纯语音绘图工具。按原始要求拆分:

- 纯语音绘图闭环: 已完成基础闭环, 前端没有鼠标拖拽绘图工具, 但仍需要无鼠标端到端验收。
- 指令理解准确性与容错: 高频规则指令和部分复杂模板已可用, 多主体复杂场景已能追问或进入 Drawing Agent 规划, 但缺少真实语音评测集。
- 响应延迟: 已记录 ASR、规划、执行和端到端指标, 并新增命令日志 P50/P75/P95 统计接口, 但没有真实音频样本、流式 ASR 和稳定 SLO。
- 复杂指令拆解: 已支持房子、星星、批量改色移动、对象查询 DSL、组级语义选择、高级相对关系选择、关系链组合查询、生成图复杂目标局部精修、图片内主体/区域/调整动作精修、连续图生图追改上下文继承和部分路径场景, 但开放式复杂构图仍未稳定产品化。
- 设计文档交付: 已记录计划能力、实现能力、未完成能力和原因, 本次新增阶段差距评估文档。

详细差距评估见 [docs/status/voice-drawing-gap-analysis.md](docs/status/voice-drawing-gap-analysis.md)。

## 2. 计划支持的指令能力

### 2.1 画布能力

- 新建画布
- 设置横向或竖向尺寸
- 设置背景颜色
- 保存作品版本
- 导出 PNG
- 清空画布前二次确认

### 2.2 绘图能力

- 圆形
- 矩形
- 椭圆
- 三角形
- 线条
- 箭头
- 星形
- 文本
- 多边形
- 路径
- 贝塞尔曲线
- 常见颜色词
- 常见位置词
- 常见尺寸参数

### 2.3 编辑能力

- 选择最近对象
- 修改最近对象颜色
- 加粗最近对象边框
- 移动最近对象
- 撤销
- 恢复

### 2.4 复杂指令能力

- 把一句复杂命令拆解为多个可执行操作
- 每个操作写入操作历史
- 操作失败时停止并返回错误

1. 房子主体
2. 红色屋顶
3. 蓝色门
4. 窗户1

语音绘图产品 UI 草图会通过 Drawing Agent 本地模板生成应用外框、侧边导航、顶部栏、搜索框、主内容卡片、指标卡片、趋势图和主行动按钮, 用于验证产品界面线框图类场景可以输出可编辑矢量结构。

产品团队组织结构图会通过 Drawing Agent 本地模板生成标题、负责人卡片、职能小组、执行角色和层级连接线, 用于验证层级关系类图形可以输出可继续编辑的矢量对象。

产品迭代项目排期甘特图会通过 Drawing Agent 本地模板生成时间轴、月份标签、任务行、任务条、今日线、上线里程碑和图例, 用于验证项目计划类图形可以输出可继续编辑的矢量结构。

### 3.10 已实现低置信度追问与计划解释

- `CommandPlan` 增加 `explanation` 和 `planner_source`
- 后端会为规则解析、Drawing Agent 和规则兜底统一补充计划解释
- “画一个温馨的小屋, 左边有两棵树, 右边有一条弯曲小路, 天空有三朵云”这类多主体场景会先返回追问
- “把画面改成夜晚, 保留房子的形状, 给窗户加暖黄色灯光”这类全局改造指令会先进入确认路径
- 规则层不会把 `planner_expected` 用例误执行成一个简单对象
- 前端执行计划卡片会展示置信度、计划来源、摘要、拆解步骤和确认状态

### 3.11 已实现清空画布确认链

- 首次说“清空画布”时, 后端返回 `requires_confirmation=true` 和高风险提示, 不删除任何对象
- 待确认计划写入 `voice_command_logs`, 状态为 `needs_confirmation`
- 继续说“确认清空”时, 后端读取同一作品最近的待确认 `clear_canvas` 计划并执行
- 清空操作作为一条 `operations` 历史记录写入, 逆操作保存被清空对象快照
- “撤销”会恢复被清空的对象, “恢复”会再次清空
- “取消清空”会把待确认日志标记为 `canceled`, 不执行绘图变更

### 3.12 已实现复杂编辑与生图增强入口

- 支持“把窗户改成圆形”这类语义对象形状替换
- 支持“把左边窗户改大一点”这类空间语义筛选和局部缩放
- 支持“把左边第二棵树改成黄色”“把屋顶下面的门改成绿色”“把卡片里和标题同一行的按钮改成绿色”“把所有暖色小物件向上移动一点”“把整个沙发向右移动一点”这类对象查询 DSL 和组级编辑
- 新增 `replace_shape` 和 `replace_shape_many` 操作, 可写入历史并支持撤销恢复
- 新增 `image` 绘图对象类型, 前端 SVG 画布可以渲染图片对象
- 新增 `generate_image_asset` 计划类型, 执行前会转换为 `add_object image`
- 新增文字转图片 Provider 脚手架, 默认 `placeholder` Provider 用于验证链路
- 支持通过 `AI_PAINTING_IMAGE_PROVIDER=http` 接入外部文字转图片模型服务
- 新增 `polish_image_asset` 计划类型, 用于“精修我的图片”“丰富当前画面”等图生图精修指令
- 前端会把当前 SVG 画布导出为 PNG data URL, 后端再把图片和提示词提交给图生图模型
- 支持 OpenAI 兼容图像生成和图像编辑接口, 可通过中转站优先调用 `gpt-image-2`
- 支持 OpenAI 官方图片 API 作为中转站失败后的备用 Provider
- 精修结果会作为覆盖全画布的 `image` 对象加入作品, 原始矢量源对象保留在下层并可撤销回退
- 支持“把人物肖像的眼睛精修一下”“把右边那张生成图的天空精修一下”“把第二张生成图精修一下”“把右边那个人的眼睛调亮”这类生成图局部精修, 后端按 `prompt_contains`、`position`、`rank` 或 `corner` 找到源图片对象, 合并源图 `prompt`、目标主体、目标区域、调整动作和用户新指令后提交给图生图 Provider
- 支持“继续把他的头发柔和一点”这类连续追改, 当最新源图是上一轮精修结果时, 后端会继承上一轮 `source_prompt` 和 `target_subject`, 避免把上一轮合成提示词当作原始提示词继续堆叠

### 3.13 已实现渲染策略路由

后端新增 `render_strategy` 分类器, 在规则解析前判断指令更适合程序生成、生图模型还是图生图精修。

程序生成优先用于位置、文字、关系线和后续编辑要求高的图形, 例如甘特图、泳道图、流程图、UML 图、ER 图、系统架构图、组织结构图、普通小房子、草地、太阳、树、简单场景组合、海报草稿版式和手抄报草稿版式。

生图模型优先用于画面美观、风格统一和艺术细节要求高的图形, 例如水墨画、二次元动漫人物、写实插画、概念场景图、复杂艺术海报、商业视觉图、儿童插画、国风插画和科幻场景。

图生图精修优先用于“精修图片”“丰富当前画面”“风格转换”和生成图局部追改等指令。系统会保留原始矢量对象或源图片对象, 把精修结果作为覆盖式图片层写入历史。

当前策略已经接入 `parse_command` 和 Drawing Agent 路由保护: 已经被规则层判断为 `generate_image_asset` 或 `polish_image_asset` 的计划不会再被 Agent 模板抢走。

## 4. 暂未完成能力与原因

- ASR 真机评测: 已接入小米 MiMo ASR 和本地 ASR 适配层, 但仍需要真实账号、真实麦克风和方言噪声样本做稳定性评测
- 多对象语义选择: 已支持按颜色、图层、语义标签、排序位置、相对位置、靠近关系、遮挡关系、包含关系、同一行/同一列、层级前后、关系链组合、颜色温度、小物件筛选、生成图复杂目标选择、图片内主体引用和组级扩展的第一版, 但组内相邻对象、路径附近对象和跨领域关系模板仍需继续扩展
- 图层编组: 已有 `group_id` 和 `layer_id` 元数据, 并已支持组级语义选择和批量变换, 但还没有完整的图层面板、取消编组和组内排序策略
- 旋转对象: 当前几何模型尚未统一 transform 字段
- 渐变和阴影: 第一阶段先保证基础矢量样式稳定
- 路径自由编辑: 已能通过语音创建和移动路径对象, 但还没有逐点调整、路径布尔运算和语音控制曲率
- 开放式复杂场景自动生成: 小屋、人物肖像和客厅等高频模板已能执行, 但开放描述的真实自动拆解仍依赖 Drawing Agent 规划器和后续评测
- Mermaid / PlantUML 结构图执行器: 当前已落地本地流程图模板, 但尚未支持 Mermaid 或 PlantUML 文本语法解析和渲染
- 真实文字转图片模型: 已实现 Provider 脚手架和图片对象链路, 但当前重点转向图生图精修, 默认文生图 Provider 仍只是占位图
- 真实图生图模型效果: 已接入 OpenAI 兼容中转站优先和 OpenAI 官方备用调用链, 但尚未用真实画布样本验证出图质量、延迟和失败返回结构
- SVG 后端导出: 当前 PNG 导出由前端基于 SVG 转换完成, 后端导出需要额外渲染依赖
- 方言和中英混合: 需要更强的 ASR 和 NLU 评测集
- 用户自定义词表: 需要用户配置持久化和冲突处理

## 5. 语音识别方案

当前采用“后端 ASR 优先, 浏览器识别兜底”的分层方案:

1. 前端通过 Web Audio API 采集麦克风音频。
2. 前端用简单能量阈值和静音窗口切分单条语音指令。
3. 前端将音频编码为 `audio/wav` data URL, 提交到后端 `/api/asr/transcribe`。
4. 后端按 `AI_PAINTING_ASR_PROVIDERS` 顺序尝试 Provider, 默认顺序为 `xiaomi,local`。
5. 小米 Provider 调用 `mimo-v2.5-asr`, 音频通过 `input_audio.data` 字段传入, 语种默认 `zh`。
6. 本地 Provider 支持本地 HTTP 服务或本地命令模板, 作为小米 ASR 后的第一备用方案。
7. 如果后端 ASR 不可用, 前端降级为浏览器 `SpeechRecognition` 或 `webkitSpeechRecognition`。

已实现能力:

- 小米 MiMo ASR 非流式调用适配
- 本地 ASR HTTP 服务适配
- 本地 ASR 命令模板适配
- 小米 ASR 请求失败时支持短暂重试
- 后端 Provider 状态查询
- 后端 Provider 失败后返回可解释的尝试记录
- 前端录音、静音切分、WAV 编码和后端转写调用
- 语音识别改为用户点击麦克风后才开始, 页面加载不自动打开麦克风
- 前端录音触发阈值提高, 降低轻微环境声被当作指令的概率
- 执行结果使用小米 MiMo TTS 播报, 不再使用浏览器 `speechSynthesis` 作为播报方案
- Web Speech API 最终兜底

当前前端切分参数:

- 能量阈值: RMS `0.045`
- 最短语音: `480ms`
- 静音截断: `1500ms`
- 最长语音: `30000ms`

已知限制:

- 当前小米 ASR 接口按整段音频提交, 不是真正的低延迟 WebSocket 流式识别
- 前端静音切分使用能量阈值, 噪声较大时可能提前或延后截断
- 本地 ASR 已提供 Qwen3-ASR 服务脚手架和后端适配入口, 但不随项目下载大模型权重
- Web Speech API 仍受浏览器支持、系统权限、网络和系统语言影响
- 尚未用真实小米 API Key、真实麦克风和方言噪声样本做端到端评测
- 小米 TTS 需要网络和 `MIMO_API_KEY`, 真实播放效果仍需在浏览器中验收

## 6. Drawing Agent 指令架构

后端采用“规则工具优先, Drawing Agent 规划复杂场景”的混合方案:

1. 文本规范化
2. 同义词替换
3. 中文数字解析
4. 颜色词映射
5. 形状词映射
6. 方位词映射
7. 意图判断
8. 简单指令直接生成操作计划
9. 低置信度或复杂场景进入 Drawing Agent Planner
10. Agent 生成 SceneGraph v2
11. Pydantic 校验 SceneGraph
12. SceneGraph Compiler 生成受控 `CommandPlan`
13. 后端校验操作类型、对象类型、对象数量和风险等级, 通过后才交给绘图引擎执行
14. 规则兜底为多主体场景返回追问和解释, 防止误执行半成品计划

规则解析的优点是延迟低、可测试、可解释。Drawing Agent 用于补足复杂自然语言拆解能力, 但只在复杂场景启用, 避免所有指令都依赖外部模型造成延迟和成本上升。

当前 Agent 已落地:

- SceneGraph v2 schema
- SceneGraph 到 `CommandPlan` 的编译器
- classify、build SceneGraph、repair、validate、repair_with_model、compile 的 LangGraph 节点化编排
- 独立 MiMo SceneGraph 模型客户端, 支持构图生成和一次模型修复
- 本地客厅场景模板
- 本地流程图和结构图模板
- 本地信息图模板
- 本地海报模板
- 本地 UI 草图模板
- 本地组织结构图模板
- 本地甘特图模板
- 小米 MiMo-v2.5-Pro SceneGraph 规划兜底
- LangGraph 不可用或节点失败时自动回退同步路径
- Agent 规划延迟新增 `agent_planner_ms`, 旧 `llm_planner_ms` 保留用于兼容历史日志和前端

## 7. 复杂指令拆解策略

复杂指令返回 `CommandPlan`, 其中包含多个 `OperationRequest`。前端只关心最终画布和执行消息, 后端负责逐步执行和持久化。

当前策略:

- 低风险复杂指令直接执行
- 清空画布这类高风险指令先返回确认问题, 只有收到同一作品后续确认语音才执行
- 多主体场景和全局改造指令在规则层先返回追问
- 每个状态变更操作写入 `operations`
- 失败时抛出错误, 不继续执行后续操作
- Drawing Agent 规划失败时回退到规则解析结果, 不影响基础指令
- Drawing Agent 最多允许有限步骤, 防止一次语音生成过大的执行计划
- 多步骤计划在同一执行单元中提交, 失败时回滚本次计划

## 8. 延迟优化策略

当前策略:

- 简单规则解析在后端同步完成
- SQLite 写入只保存必要 JSON
- SQLite 增加按作品、类型、状态和创建时间的查询索引
- 多步骤计划减少重复 redo 栈清理和中间提交
- 前端 SVG 增量来自后端返回的最新对象列表
- 前端初始化作品时增加防重保护, 减少开发态重复 API 调用
- 语音识别、解析、执行耗时写入 `voice_command_logs.latency_json`

后续可优化:

- WebSocket 推送复杂计划执行进度
- 前端先乐观渲染低风险操作
- 更细粒度的对象增量更新

## 9. SQLite 表结构

当前后端初始化以下表:

- `artworks`: 作品基础信息
- `artwork_versions`: 作品版本快照
- `drawing_objects`: 当前绘图对象
- `operations`: 可撤销操作历史
- `voice_command_logs`: 语音指令解析和执行日志

SQLite 连接层会执行 `PRAGMA foreign_keys = ON`, 保证外键约束生效。

当前索引:

- `idx_drawing_objects_artwork_z`
- `idx_drawing_objects_artwork_type`
- `idx_drawing_objects_artwork_layer`
- `idx_drawing_objects_artwork_group`
- `idx_operations_artwork_status_created`
- `idx_voice_logs_artwork_created`

`drawing_objects` 元数据字段:

- `layer_id`: 对象所在图层, 默认 `base`
- `group_id`: 对象所属语义组
- `semantic_tags_json`: 对象语义标签列表
- `transform_json`: 为旋转、缩放、倾斜等后续变换预留

图生图精修也通过语音命令入口触发。前端在识别到精修类指令后, 会把当前画布截图放入 `canvas_image_data_url`, 后端把 `polish_image_asset` 转换为 `add_object image` 后执行。若指令能通过提示词、空间位置、第 N 张或四角位置定位到已有生成图片, 后端优先使用该图片对象的 `src` 和原始 `prompt`。若指令描述图片内主体、区域和调整动作, 例如“右边的人”“眼睛”“调亮”, 后端会把 `target_subject`、`target_region` 和 `adjustment` 写入提示词与结果图片元数据。连续追改语境下, 后端会从上一轮精修图片继承原始 `source_prompt` 和目标主体, 但不会声称已经具备视觉分割或 mask 编辑。当前 Provider 顺序为中转站优先, OpenAI 官方 API 备用。

延迟指标:

- ASR 响应返回 `metrics.total_ms`、`audio_bytes`、`attempt_count`、`successful_provider` 和 `fallback_count`
- 命令执行响应返回 `metrics.rule_parse_ms`、`llm_planner_ms`、`planner_total_ms`、`execute_ms`、`total_ms`、`llm_attempted`、`llm_succeeded`、`fallback_used` 和 `planner_source`
- 命令延迟指标会同步写入 `voice_command_logs.latency_json`
- 延迟统计接口返回最近日志的平均值、P50、P75、P95、最大值、状态计数和规划来源计数

## 11. 前端交互设计

前端第一屏就是绘图工作台, 包含:

- 左侧全高度 SVG 画布
- 右侧控制台标题和 icon-only 工具按钮
- 右侧监听状态
- 识别文本
- 最近一次 ASR、规划、执行和端到端延迟指标
- 操作历史

前端没有绘图工具栏、拖拽交互或键盘快捷绘图路径。

画布渲染技术选择:

- 当前 `CanvasStage` 使用 SVG 渲染结构化绘图对象, 支持对象级语义标签、图层、分组、撤销恢复和前端 PNG 导出
- 当前“只能画简单图形”的主要瓶颈不是 SVG 渲染, 而是自然语言到结构化对象计划的拆解能力仍偏模板化
- HTML Canvas 适合像素级笔刷、海量对象性能优化和滤镜后处理, 但会削弱对象级可编辑性和语音精确修改能力
- 近期建议保留 SVG 作为主编辑层, 同时把图生图结果、纹理和背景作为 `image` 对象叠加
- 中期可以引入混合渲染: SVG 负责可编辑矢量对象, Canvas 或 OffscreenCanvas 负责复杂预览、滤镜、笔刷和大图导出
